JSON marshaling and unmarshaling are processes used in computer programming to convert data between the JSON (JavaScript Object Notation) format and a programming language's data structures.
JSON Marshaling:
JSON marshaling, also known as serialization or encoding, is the process of converting a data structure or object in a programming language into a JSON-formatted string. This is typically done when you want to send or store data in a format that can be easily transmitted over the internet or saved in a file. The JSON format is text-based and human-readable, making it widely used for data interchange.
During JSON marshaling, the data in the programming language's data structure is transformed into a corresponding JSON representation. For example, in most programming languages, a dictionary or map might be converted into a JSON object, while an array or list might be transformed into a JSON array.

JSON Unmarshaling:
JSON unmarshaling, also known as deserialization or decoding, is the reverse process of JSON marshaling. It involves converting a JSON-formatted string back into a data structure or object in the programming language. This is commonly used when receiving JSON data from an external source, such as a web API, and you need to work with that data in your application.
During JSON unmarshaling, the JSON-formatted string is parsed, and its data is used to reconstruct the original data structure in the programming language.

For example, in most programming languages, a dictionary or map might be converted into a JSON object, while an array or list might be transformed into a JSON array.

JSON Unmarshaling:
JSON unmarshaling, also known as deserialization or decoding, is the reverse process of JSON marshaling. It involves converting a JSON-formatted string back into a data structure or object in the programming language. This is commonly used when receiving JSON data from an external source, such as a web API, and you need to work with that data in your application.
During JSON unmarshaling, the JSON-formatted string is parsed, and its data is used to reconstruct the original data structure in the programming language.

Many programming languages provide built-in functions or libraries for JSON marshaling and unmarshaling, making it straightforward to work with JSON data in a seamless manner. For example, in Python, you can use the json module to perform JSON serialization and deserialization. In JavaScript, you can use JSON.stringify() for marshaling and JSON.parse() for unmarshaling. Similarly, most modern programming languages have similar capabilities for working with JSON data.

Css Box Model 
Within CSS; every element within on a web page is treated as a rectangular box. The CSS Box Model is a set of rules which determine how to caluclate the
dimensions of that box and how the parts/sections of the box - content, padding, border and margin - interact with each other.
Box Model is composed of four layers, radiating from the centre
1. Content: Innermost part where the actual text, image, media etc appears 
2. Padding: Surrounding Area of the content which creates the internal space inside of the box (from within since internal below external space)
3. Border: The line which surrounds the padding area can can b styled, colored, given a thinkness 
4. Margin: The outermost layer which surrounds the margin and provides an external space outside the box 
  ex 
      .my-box {
        /* Content Dimensions */
        width: 200px;
        height: 100px;

        /* Padding */
        padding: 20px; /* Adds 20px of space inside, on all sides */

        /* Border */
        border: 5px solid blue; /* Adds a 5px thick blue border on all sides */

        /* Margin */
        margin: 30px; /* Adds 30px of space outside, on all sides */

        /* Just for better visibility */
        background-color: lightyellow;

Virtual DOM - React's RAM ? 
  is react's in memory representation of the actual dom. Its a light-weight js object that mirrors the structure of the real dom. 
  How the Virtual DOM Works Better explained above in the jsx section 

Initial Render: React creates a Virtual DOM representation when a component first renders
State/Props Change: When state or props update, React re-renders and creates a new Virtual DOM
Diffing Algorithm: React compares (diffs) the new Virtual DOM with the previous one to identify what changed
Reconciliation:
  Reconciliation in React
  React calculates the minimal set of DOM updates needed
    changes in state / prop or both
    diffing 
    update the dom 
    commitPhase
Batch Updates: React updates only the changed parts in the actual DOM, not the entire tree

Why Virtual DOM Matters

Predictability: The rendering process is deterministic and follows clear rules
Abstraction: Developers don't need to manually manage DOM updates; React handles it automatically
Performance: Direct DOM manipulation is slow. The Virtual DOM allows React to batch changes and update the real DOM efficiently
Cross-platform: The Virtual DOM concept enables React Native and other frameworks to target different platforms

Controlled Components & Un-controlled components in ReactJs
  form inputs & other interactive elements can be managed in two main ways: controlled or uncontrolled components. This disction helps in handling user input, validation 
  & state synchronization. 
  Aspect,             Controlled Components,      Uncontrolled Components
  Value Management,React state (via value prop),  DOM (via ref)
  State Updates,    On every change (onChange),"On demand (e.g., submit)"
  Example Props,    value={state} + onChange={setState},ref={inputRef}
  Re-renders,       Frequent (per keystroke),None on input changes
  Best For,          Interactive UIs, validation, dynamic forms","Simple forms, performance-critical inputs"
  Testing,            Easy (mock state changes),Harder (mock refs/DOM)
  Code Complexity,  Higher (state + handlers),Lower (minimal setup)
  React Philosophy   Everything is state (preferred Let DOM handle it"" (legacy-friendly)"

  ensuring app's responsivness EVEN DURING HEAVY UPDATES

  this lead to dropping of frames or UI freezes when the component tree was large. 
  BUT with concurrent rendering react can: 
    o pause rendering when something urgent happens(user clicking)
    o abort out-dated renders(user navigating mouse away before rendering finishes)
    o Resume rendering later, 
    o prioritize more important updates earlier(like typing vs background data refresh)
    
    In essense React >=v18 schedules rendering tasks intelligently, like an OS SCHEDULAR, so ur app feels fast even under load.

  useLayoutEffect()    
    useLayoutEffect is a React hook that runs synchronously immediately after React has performed all DOM mutations BUT BEFORE the browser paints the screen
    NOTE: Never access window, document, or layout in useLayoutEffect during SSR.
      Use useEffect or guard with typeof window !== "undefined".
    This hook will execute the containing callBack of course after the render BUT BEFORE anything was displayed/painted on the screen for the user to see. 
    import { useEffect, useLayoutEffect, useRef, useState } from "react";

function Box() {
  const ref = useRef<HTMLDivElement>(null);
  const [height, setHeight] = useState(0);

  // Flicker version
  useEffect(() => {
    if (ref.current) setHeight(ref.current.offsetHeight);
  }, []);

  // No flicker
  useLayoutEffect(() => {
    if (ref.current) setHeight(ref.current.offsetHeight);
  }, []);

  return (
    <div ref={ref} style={{ height: "200px", background: "salmon" }}>
      Height: {height}px
    </div>
  );
}

Partials in Sass are separate files that hold segments of CSS code. These are imported and used in other Sass files. This is a great way to group similar code into a module to keep it organized.

Names for partials start with the underscore (_) character, which tells Sass it is a small segment of CSS and not to convert it into a CSS file. Also, Sass files end with the .scss file extension. To bring the code in the partial into another Sass file, use the @import directive.

For example, if all your mixins are saved in a partial named "_mixins.scss", and they are needed in the "main.scss" file, this is how to use them in the main file:
 
@import 'mixins'
@import 'variables' 
Note that the underscore and file extension are not needed in the import statement - Sass understands it is a partial. Once a partial is imported into a file, all variables, mixins, and other code are available to use.
Sass has a feature called extend that makes it easy to borrow the CSS rules from one element and build upon them in another.
