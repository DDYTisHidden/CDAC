How does Spring Boot Auto-Configuration actually work under the hood?Answer: Spring Boot uses the @EnableAutoConfiguration annotation (included in @SpringBootApplication). During startup, Spring searches for conditional configuration classes declared in:  META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports (Spring Boot 3.0+)META-INF/spring.factories (Spring Boot 2.x)  It evaluates annotations like @ConditionalOnClass, @ConditionalOnMissingBean, and @ConditionalOnProperty to determine whether to register a bean automatically.  2. Why does @Transactional sometimes fail or get bypassed?Answer: @Transactional uses Spring AOP proxies (Dynamic Proxies or CGLIB). It fails to apply when:Self-Invocation: A method calls another @Transactional method within the same class (bypasses the proxy).Non-Public Methods: Spring AOP proxies only intercept calls to public methods.  Unchecked vs. Checked Exceptions: By default, transactions roll back on unchecked exceptions (RuntimeException, Error), but not on checked exceptions unless specified via @Transactional(rollbackFor = Exception.class).  3. What is the difference between Constructor Injection and Field Injection, and why is Field Injection discouraged?Answer:Constructor Injection (Recommended): Makes dependencies explicit, allows instances to be immutable (final), and facilitates unit testing without needing Spring reflection/mocks.  Field Injection (@Autowired on field): Makes unit testing difficult, hides dependencies, and risks NullPointerException outside the Spring container lifecycle.  4. What causes LazyInitializationException in Spring Data JPA, and how do you resolve it?Answer: It occurs when accessing a lazily loaded (FetchType.LAZY) collection or association outside an active Hibernate session/transaction (e.g., in a Controller layer after the Service transaction has ended).  Solutions:Use DTO projections instead of returning entities directly.  Fetch relationships via JOIN FETCH in JPQL or use @EntityGraph.  Avoid enable_lazy_load_no_trans=true in production as it leads to severe performance degradation.5. How do you solve the N+1 SELECT query problem in Spring Data JPA?Answer: When fetching $N$ parent records, JPA triggers 1 query for the parents and $N$ additional queries to fetch their child associations.  Solutions:Use JOIN FETCH in custom JPQL/HQL queries.Use @EntityGraph(attributePaths = {"children"}).Use @BatchSize to fetch records in batches rather than individually.⚡ Part 2: Advanced Java Core & JVM Internals6. What are Virtual Threads (Java 21 / Project Loom), and how do they differ from Platform Threads?Answer:Platform Threads: 1:1 mapping with OS threads. They are expensive to create (~1MB stack memory) and limited in count (thousands).Virtual Threads: Managed directly by the JVM (lightweight, ~few KBs). Million+ virtual threads can run concurrently over a small pool of carrier OS threads. They are ideal for high-throughput, blocking I/O bound workloads.7. What is the difference between CompletableFuture and ExecutorService?Answer:ExecutorService provides a thread pool to execute asynchronous tasks, returning a Future (which requires blocking calls like .get() to retrieve results).CompletableFuture allows non-blocking reactive-style chaining, combining multiple async stages (thenApply, thenCompose, exceptionally), handling asynchronous pipelines declaratively without blocking threads.8. How does the Java Memory Model handle volatile, synchronized, and ReentrantLock?Answer:volatile: Guarantees visibility across threads by flushing reads/writes directly to main memory (disables CPU cache buffering and instruction reordering for that variable). It does not guarantee atomicity for compound operations (e.g., i++).synchronized: Guarantees both visibility and atomicity using intrinsic monitor locks.ReentrantLock: Offers advanced concurrency features over synchronized, such as fair locking, interruptible lock acquisition, and explicit Condition variables.9. What are Records, Sealed Classes, and Pattern Matching in modern Java?Answer:Records (Java 16): Immutable data carrier classes that automatically generate getters, equals(), hashCode(), and toString().Sealed Classes (Java 17): Restricts which classes or interfaces can extend or implement them using the permits keyword.Pattern Matching (Java 17–21): Simplifies type casting in instanceof checks and switch statements, enabling clean functional/algebraic data handling.🏗️ Part 3: System Design & Microservices Scenarios10. How do you handle distributed transactions across Microservices in Spring Boot?Answer: Avoid 2-Phase Commit (2PC) in cloud-native microservices due to low performance and blocking behavior.Preferred Approaches:Saga Pattern: Break transactions into local transactions per service. Use Orchestration (central coordinator) or Choreography (event-driven via Kafka/RabbitMQ) with compensating transactions to roll back state on failure.Outbox Pattern: Guarantee event publication reliability by writing events to a DB "Outbox" table in the same local transaction, then streaming them via CDC (e.g., Debezium) to event brokers.11. Validate JWT signatures statelessly at the gateway or downstream services, and pass identity context via HTTP headers (or propagate Spring Security context using SecurityContextHolder).Would you like to dive deeper into any specific area—such as Spring Security internal architecture, Kafka integration, or hands-on coding/scenario-based problems?
