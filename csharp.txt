 An event is a member (not a type) built on top of a delegate that provides a controlled, publish-subscribe mechanism. 
 It restricts how the underlying delegate can be used from outside the declaring class: external code can only += (subscribe) or -= (unsubscribe), but cannot directly invoke the delegate or overwrite its invocation list (=), and cannot see/clear other subscribers' handlers.
 Essentially: an event is a delegate with access-restricted invocation, following the observer/publish-subscribe design pattern.
    class Publisher
    {
        // Declares an event based on a delegate type
        public event EventHandler<string> OnMessage;

        public void RaiseMessage(string msg)
        {
            // Only the declaring class can invoke it
            OnMessage?.Invoke(this, msg);
        }
    }

    class Subscriber
    {
        public void Handle(object sender, string msg) =>
            Console.WriteLine($"Received: {msg}");
    }

    var pub = new Publisher();
    var sub = new Subscriber();

    pub.OnMessage += sub.Handle;  // ✅ subscribing is allowed
    // pub.OnMessage(pub, "hi"); // ❌ compile error — can't invoke from outside
    // pub.OnMessage = null;     // ❌ compile error — can't overwrite from outside

    pub.RaiseMessage("Hello!");   // triggers all subscribers

A) Kestrel is an In-Process compiler used to convert Razor views into executable C# classes at runtime.

B) Kestrel is a lightweight, cross-platform web server enabled by default, which can run an ASP.NET Core app independently without external servers like IIS or Nginx.

Correct Answer: B
Reason: Thinking ASP.NET Core requires a heavy enterprise web server like IIS to function, or assuming Kestrel does everything alone in production.
Why the others look tempting but are wrong:

Option C is the big trap. In production, it is recommended to put Nginx/IIS in front of Kestrel as a reverse proxy for security and load balancing, but it is not a strict system requirement. Kestrel is fully capable of listening directly to internet traffic on its own.

Option A plays on keywords, trying to trick you into confusing a web server with a compilation engine.

Yes. **LINQ is closely related to the collection framework, but it is not itself a collection.**

Think of it like this:

* **Collections** → Store data (`List<T>`, `Dictionary<TKey,TValue>`, `HashSet<T>`, etc.)
* **LINQ** → Queries and transforms data stored in collections (or other data sources like XML, databases, etc.)

So in the PGCP-AC MS.NET module:

* **Collections** and **LINQ** are usually treated as **separate syllabus topics**, although LINQ is most commonly used with collections.

Example:

```csharp
List<int> nums = new List<int> { 10, 20, 30, 40, 50 };

var result = nums.Where(n => n > 25)
                 .OrderByDescending(n => n);

foreach(var x in result)
    Console.WriteLine(x);
```

Here:

* `List<int>` is the collection.
* `Where()` and `OrderByDescending()` are LINQ extension methods.
