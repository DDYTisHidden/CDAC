Definition and Characterization: Deadlock is a permanent state where processes are waiting indefinitely for resources held by each other. Know examples (e.g., dining philosophers).
Necessary Conditions (Coffman's Conditions): All four must hold – Mutual Exclusion (resources can't be shared), Hold and Wait (process holds resources while waiting for others), No Preemption (resources can't be forcibly taken), Circular Wait (cyclic dependency).
Resource Allocation Graph (RAG): Directed graph with processes (circles), resources (squares), request edges (process to resource), assignment edges (resource to process). Deadlock if cycle exists and no way to reduce (for single-instance resources).
Deadlock Prevention: Eliminate one condition – e.g., allow sharing (no mutual exclusion), request all resources at start (no hold and wait), allow preemption, or order resources hierarchically (no circular wait).
Deadlock Avoidance: Runtime check to ensure safe state. Know Banker's Algorithm: Safe state (exists a sequence where all processes can finish), Need Matrix (Max - Allocation), Safety Algorithm (simulate allocation), Resource-Request Algorithm. Practice numericals with small data.
Deadlock Detection: Periodic check for cycles in RAG or using a variant of Banker's. For multiple instances, use wait-for graph.
Deadlock Recovery: Once detected, recover via process termination (victim selection: least cost, youngest, etc.), resource preemption (rollback), or ignore (ostrich algorithm).
Related Concepts: Starvation (indefinite postponement, e.g., due to priority), Livelock (processes change states but no progress), Thrashing (related to paging but sometimes confused).
Handling in Real OS: Examples like Windows/Linux deadlock detection tools, avoidance in practice (rare due to overhead).
Practice Tips: Solve 50+ MCQs from GeeksforGeeks, GATE papers (1990-2020), and CDAC-specific samples. Focus on why options are wrong. For numericals, practice 5-10 Banker's problems.

  Thread Fundamentals:

    Process vs. Thread: Understanding that a process is a heavy-weight unit of execution, while a thread is a light-weight unit.

    Thread Lifecycle: The various states a thread can occupy (e.g., ready, running, blocked).

    Shared vs. Private Resources: What threads share within a process (code, data, heap, files) versus what is unique to each thread (program counter, registers, stack).

    Multithreading Models:

    Many-to-One: Multiple user threads mapped to a single kernel thread.

    One-to-One: Each user thread maps to a separate kernel thread.

    Many-to-Many: Multiplexing many user-level threads to a smaller or equal number of kernel threads.

    Synchronization & Mutual Exclusion (Crucial):

    Critical Section Problem: Ensuring only one thread accesses a shared resource at a time.

    Synchronization Primitives: Deep understanding of Mutex locks, Semaphores (counting vs. binary), and Condition Variables.
    Race Conditions: Understanding how concurrent access to shared data leads to inconsistencies.
    Deadlock Management:
    Coffman Conditions: The four necessary conditions for deadlock: Mutual Exclusion, Hold and Wait, No Preemption, and Circular Wait.
    Prevention, Avoidance, and Detection: Knowing the differences between techniques like the Banker's Algorithm (avoidance) and simply detecting cycles in a wait-for graph.
    Performance & System Design:
    Concurrency vs. Parallelism: Concurrency is about managing multiple tasks (progress is made on more than one, but not necessarily at the same time), whereas parallelism is the simultaneous execution on multiple cores.
    Context Switching: Why switching between threads is "cheaper" (faster) than switching between processes.

    Good question — and the "why is Java even here" confusion makes sense once you see how the OS module is actually structured at CDAC. Here's what's going on, based on the current syllabus and CCEE prep materials.

## Why Java shows up in the OS/DevOps module

This module isn't really "just OS" — it bundles **Linux/OS fundamentals + shell scripting + Git + DevOps toolchain (Docker, Jenkins) + Selenium-based test automation** into one paper. Java's inclusion isn't because they're testing you on core Java/J2EE (that's a separate subject with its own CCEE paper) — it's because the Selenium module requires environment setup with Java, an IDE, Maven, and browser drivers as prerequisites for writing WebDriver scripts, and Selenium with TestNG involves annotations, assertions, suites, and execution basics — all of which are Java-syntax-dependent.

So Java in this paper = **"enough Java to read and write a Selenium/TestNG script"**, not enterprise Java. Past-student CCEE notes confirm the pattern across papers: Java questions focus on OOPs concepts, language features, and interfaces and methods of predefined classes — and in the OS/DevOps paper specifically, that knowledge gets tested *through* the Selenium lens.

## Specific Java aspects likely to be asked

Given the syllabus ties Java tightly to Selenium/TestNG, expect MCQs clustered around these areas:

**1. OOP fundamentals (conceptual, not code-heavy)**
- Interfaces vs abstract classes — important because `WebDriver` itself is an **interface**, and questions often test "why is WebDriver an interface" or "which classes implement WebDriver" (ChromeDriver, FirefoxDriver, etc.)
- Polymorphism/overriding — since browser-specific driver classes override WebDriver methods
- Encapsulation, inheritance basics — standard conceptual MCQs

**2. Exception handling**
- try-catch-finally, checked vs unchecked exceptions
- This matters a lot because Selenium scripts constantly deal with `NoSuchElementException`, `TimeoutException`, `StaleElementReferenceException` — expect questions on exception hierarchy and handling patterns in this exact context

**3. Collections framework**
- List, ArrayList, Iterator — because `findElements()` in Selenium returns a `List<WebElement>`, and looping through elements (dropdowns, tables, multiple links) is a bread-and-butter Selenium task
- Expect at least one question testing whether you know `findElement()` returns a single object vs `findElements()` returns a List

**4. Annotations (TestNG-specific)**
- `@Test`, `@BeforeMethod`, `@AfterMethod`, `@BeforeSuite`, `@BeforeClass` and their execution order
- This is a very common MCQ type: "what is the execution order given these annotations"

**5. Access modifiers, static, this, method overloading**
- Standard Java basics, asked as short conceptual/code-snippet questions

**6. String handling**
- String vs StringBuilder, common String methods — sometimes tested because Selenium scripts manipulate element text/attributes constantly

**7. Architecture-level Selenium-Java questions**
- Not "Java architecture" (JVM/JRE/JDK) in depth, but *WebDriver architecture*: client-server model, JSON Wire Protocol/W3C protocol basics, how driver communicates with browser — this is where "architecture" questions actually live in this paper, dressed up as Java/Selenium integration rather than pure Java internals

## What NOT to over-invest in
Deep JVM internals, multithreading, generics edge cases, Java 8+ streams/lambdas — these belong to the separate core-Java/J2EE paper, not this one. In the OS/DevOps paper, Java is a **supporting language for tooling**, so don't burn disproportionate time on advanced Java when Linux commands, shell scripting, and Docker/Jenkins/Git carry more of the mark distribution.

Since you want to leave no stone unturned but time is presumably limited before CCEE:
