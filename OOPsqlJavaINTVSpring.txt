	String interning in Java is a memory optimization technique where identical string literals are stored in a special memory area called the String Pool (or String Constant Pool), allowing

		When you create a string literal, the JVM checks if an identical string already exists in the string pool:

		If it doesn't exist, it creates a new String object in the pool

Differences between a shallow copy and a deep copy in Java Collections? 
	The difference between shallow copy and deep copy in Java Collections is fundamental to understanding how object references and memory work. Let me explain both concepts with detailed examples.What is Shallow Copy?A shallow copy creates a new collection object, but the elements inside the collection are not duplicated. Instead, both the original and copied collections share references to the same objects.What is Deep Copy?A deep copy creates a new collection object and creates new copies of all the objects contained within it. The original and copied collections contain completely independent objects.
	Shallow Copy:
	Original List: [obj1, obj2, obj3] -----> Objects in memory
	Copied List:   [obj1, obj2, obj3] -----> Same objects in memory

	Deep Copy:
	Original List: [obj1, obj2, obj3] -----> Original objects in memory
	Copied List:   [obj1', obj2', obj3'] ----> New objects in memory (copies)
	Aspect            Shallow  Copy          Deep Copy
	Objects           Shares references    Creates new objects
	Memory Usage    Low                    High
	Performance      Fast                   Slower
	Independence     Changes affect both     Changes are independent
	Use CasesImmutable objects, performance-criticalMutable objects, data isolationImplementationnew ArrayList<>(original)Custom copy constructors, serializationSafetyRisk of unintended modificationsSafe from external modifications
How does an ArrayList handle situations where the number of elements exceeds the initial capacity?
	ArrayList in Java handles capacity overflow through a process called dynamic resizing or capacity expansion. When the number of elements exceeds 
	the current capacity, ArrayList automatically creates a larger internal array and copies all existing elements to it. Let me explain this process in detail.

		public class Employee implements Comparable<Employee> {
	    private String department;
	    private String name;
	    private int salary;
	
	    public Employee(String department, String name, int salary) {
	        this.department = department;
	        this.name = name;
	        this.salary = salary;
	    }
	
	    @Override
	    public int compareTo(Employee other) {
	        // Primary: department
	        int deptComparison = this.department.compareTo(other.department);
	        if (deptComparison != 0) {
	            return deptComparison;
	        }
	
	        // Secondary: salary (descending)
	        int salaryComparison = Integer.compare(other.salary, this.salary);
	        if (salaryComparison != 0) {
	            return salaryComparison;
	        }
	
	        // Tertiary: name
	        return this.name.compareTo(other.name);
	    }
	
	    // Getters and toString()
	    public String getDepartment() { return department; }
	    public String getName() { return name; }
	    public int getSalary() { return salary; }
	
	    @Override
	    public String toString() {
	        return String.format("Employee{dept='%s', name='%s', salary=%d}", 
	                           department, name, salary);
	    }

Java Streams provide a powerful and flexible way to process sequences of elements, allowing for efficient and concise manipulation of collections and other data sources. 
  They enable operations such as filtering, mapping, and reducing in a functional and declarative style.

What it is: A Stream is an abstraction that represents a sequence of elements supporting sequential and parallel aggregate operations. Streams are part of the Java 8 java.util.stream package.
it runs how many times -> 																																																				once 
How it works: Streams support a variety of operations such as filtering, mapping, and reducing on data, providing a fluent API for processing sequences of elements in a functional style.
Example: Processing a list of integers to find the sum of all even numbers.
Simple Analogy: Like a pipeline where you can pass data through various stages (filters, transformations) to get the desired result.
Key Concepts
Stream Operations
Intermediate Operations: Operations that transform a stream into another stream (e.g., filter, map). These are lazy, meaning they are not executed until a terminal operation is invoked.
Terminal Operations: Operations that produce a result or a side effect (e.g., collect, forEach). These trigger the execution of the intermediate operations.
Example:
java List\<Integer> numbers \= Arrays.asList(1, 2, 3, 4, 5); int sum \= numbers.stream() .filter(n -> n % 2 \=\= 0) .mapToInt(Integer::intValue) .sum();

come back to Streams API

Explanation: A set of classes and interfaces in the java.util.stream package that provides methods for working with streams.
Example: Stream, IntStream, LongStream, DoubleStream, and utility classes such as Collectors.

Functional Style

Explanation: Streams encourage a functional programming style, where operations are specified using lambda expressions or method references.
Example:
java List\<String> names \= Arrays.asList("Alice", "Bob", "Charlie"); names.stream() .map(String::toUpperCase) .forEach(System.out::println);

Parallel Streams

Explanation: Streams can be processed in parallel to leverage multi-core processors for better performance.
Example:
java List\<Integer> numbers \= Arrays.asList(1, 2, 3, 4, 5); int sum \= numbers.parallelStream() .filter(n -> n % 2 \=\= 0) .mapToInt(Integer::intValue) .sum();
❓ How it is used?
In Data Processing
Usage: Stream operations can simplify complex data processing tasks, such as filtering, sorting, and aggregating.
Example: Processing a list of records to extract and summarize specific information.

In Collections Framework

Usage: Streams provide a more declarative way to handle collections, making code more readable and expressive.
Example: Sorting and filtering a collection of objects based on certain criteria.

In Functional Programming

Usage: Streams enable a functional programming style in Java, supporting operations with lambda expressions and method references.
Example: Mapping a list of objects to a new list with transformed values.
Summary
Java Streams: A feature in Java 8 that provides a functional approach to processing sequences of elements, supporting operations like filtering, mapping, and reducing.
Benefits: Improved readability, concise code, support for parallel processing, and functional programming style.

	import java.util.Arrays;
	import java.util.List;

	@FunctionalInterface
	interface ThrowingFunction<T, R> {
	    R apply(T t) throws Exception;
	}

	public class StreamExceptionHandling {
	    // Wrapper to handle exceptions
	    public static <T, R> java.util.function.Function<T, R> wrap(ThrowingFunction<T, R> throwingFunction) {
	        return t -> {
	            try {
	                return throwingFunction.apply(t);
	            } catch (Exception e) {
	                System.err.println("Exception: " + e.getMessage());
	                return null; // or throw a runtime exception
	            }
	        };
	    }

	    public static void main(String[] args) {
	        List<String> strings = Arrays.asList("1", "2", "abc", "3");

	        strings.stream()
	               .map(wrap(str -> Integer.parseInt(str)))
	               .forEach(System.out::println);
	    }
	}

ACID is 
	Those set of 4 fundamental properties which tests the reliability of transactions of a given database management system. 
| Property            | Meaning                                                    | Simple Example                                                                                                    |
| ------------------- | ---------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------- |
| A — Atomicity | A transaction is all or nothing.                       | If you transfer ₹500 from Account A → B, either both debit and credit happen, 
																															or none happen. No half-done state. |
| C — Consistenc | Data must stay valid according to rules.               | If total money in all accounts = ₹10,000 before transfer, it must remain ₹10,000 
																												after transfer.                  |
| I — Isolation | Transactions don’t interfere with each other.              | If two people are updating the same account, their transactions don’t clash or 
																															mix up.                            |
| D — Durability | Once saved, data stays saved — even if the system crashes. | After you get a “Transaction Successful” message, your transfer won’t vanish if the server restarts.              |

	  REST (Representational State Transfer) defines a set of architectural constraints for application programming interface which ensures scalable and maintainable web services.
      Your API follows these by ensuring:
      
      Use of Representations:
      Resources (like NewsArticle) are represented in standard formats (JSON, XML). Clients interact through these representations, not the database entities themselves.

      Client–Server Separation:
      The client (frontend) and server (backend) are independent — frontend only needs to know endpoints and response formats, 
	  not the internal bussiness logic of the server for generating the response.
      
      Statelessness:
      No session state is stored on the server — the server doesn’t “remember” previous requests from the same client.
      Each HTTP request should have sufficient information for the server to generate a response.
	  Server should not be like - aadhi request abhi laa do mei yaad rakhunga baaki ki baad mei le aa naa   
      
      Uniform Interface:
      A consistent way of interacting with the backend resources using standard HTTP methods (GET, POST, PUT, DELETE).

      GET /api/v1/news → retrieve all news
      
      POST /api/v1/users → create a new user
      
      DELETE /api/v1/history/{id} → delete a history record
      
      Layered System:
      The client doesn’t need to know if it’s communicating directly with the server or via proxies, caches, or gateways.
      
      Cacheability:
      Responses must define whether they are cacheable to improve performance and scalability.

✅ Challenges - Complex filtering with GROUP BY and HAVING
	Skills: Complex GROUP BY + HAVING + COUNT subquery
	Description: For each hacker, count distinct challenges they created. Exclude hackers who have the same count as someone else unless that count is the maximum.
	SELECT h.hacker_id, h.name, COUNT(c.challenge_id) AS cnt
	FROM Hackers h
	JOIN Challenges c ON h.hacker_id = c.hacker_id
	GROUP BY h.hacker_id, h.name
	HAVING cnt = (
	    SELECT MAX(temp.cnt)
	    FROM (SELECT hacker_id, COUNT(challenge_id) AS cnt 
	          FROM Challenges GROUP BY hacker_id) temp
	)
	OR cnt IN (
	    SELECT t.cnt
	    FROM (SELECT hacker_id, COUNT(challenge_id) AS cnt 
	          FROM Challenges GROUP BY hacker_id) t
	    GROUP BY t.cnt
	    HAVING COUNT(t.hacker_id) = 1
	)
	ORDER BY cnt DESC, h.hacker_id;

Mocking 
	Mocking is a process of creating the objects that act as the mock or clone of the real objects. In other words, mocking is a testing technique where mock objects are used instead of real objects for testing purposes. Mock objects provide a specific (dummy) output for a particular (dummy) input passed to it.
	The brief description of the mocking concept is given below:

	Stub: Stub objects hold predefined data and provide it to answer the calls during testing. They are referred to as dummy objects with a minimum number of methods required for a test. It also provides methods to verify other methods used to access the internal state of a stub, when necessary. Stub object is generally used for state verification.

	Fake: Fake are the objects that contain working implementations but are different from the production one. Mostly it takes shortcuts and also contains the simplified version of the production code.

	Mock: Mock objects act as a dummy or clone of the real object in testing. They are generally created by an open-source library or a mocking framework like Mockito, EasyMock, etc. Mock objects are typically used for behavior verification.


	We’ll be using Mockito as a mocking framework to mock the dependencies of the services. Let’s look at an example here:

What is a Data Repository?
NOTE: keep scrolling downwards since Advantages of repository pattern(repositoryNameImpl.class is new to me) are enlisted way down below
We know that the Data Store stores the data that is required for the application. The Data Repository abstracts the Data Store and enables your business logic to define read and write operations at a logical level. It does that by providing a set of methods to read, persist, update, remove and other relevant operations for entity objects from the underlying Data Store.

This means that the Data Store can change from, say, a simple Data Structure (Class object), to a database or an online storage bucket and the application wouldn’t be affected by this.


This is also known as the Repository Pattern.

Implement the Data Repository using Repository Pattern
This is how we can go about Implementing the Repository.

An interface defines the repository with all logical read and write operations for a specific data object. You can see an example of such a repository interface below:


public interface BookRepository {

   Book getBookById(Long id);

   Book getBookByTitle(String title);

   Book saveBook(Book b);

   void deleteBook(Book b);

}

The interface is implemented by one or more classes that provide Data Store specific implementations of each interface method.


A simple In Memory Data Structure based repository implementation would look like this (here, the in memory data structure, represented by the attributes in the class, is the Data Store):



public class BookRepositoryImpl implements BookRepository {

   private Map<Long,Book> bookMap = new HashMap<String,Book>() ;

   

   public BookRepositoryImpl(){}


   public BookRepositoryImpl(Map<Long,Book> bookMap ) {

       this.bookMap = bookMap;

   }

      

   @Override

   public Book getBookById(Long id) {

       return bookMap.get(id);

   }


   @Override

   public Book getBookByTitle(String title) {

     for (Map.Entry<String, Integer> e : map.entrySet()){

         if(e.getValue().getTitle().equals(title){

             return e.getValue();

         }

     }

   }


   @Override

   public Book saveBook(Book b) {

       bookMap.put(b.getId(), b);

       return b;

   }


   @Override

   public void deleteBook(Book b) {

        bookMap.remove(b.getId());

   }

}

In comparison, a simple JPA based implementation involving an actual database would look like this (here, Data Store is the database):



import javax.persistence.EntityManager;

import javax.persistence.TypedQuery;


public class BookRepositoryJPAImpl implements BookRepository {

   private EntityManager em;

   

   public BookRepositoryJPAImpl(EntityManager em) {

       this.em = em;

   }

   

   @Override

   public Book getBookById(Long id) {

       return em.find(Book.class, id);

   }


   @Override

   public Book getBookByTitle(String title) {

       TypedQuery<Book> q = em.createQuery("SELECT b FROM Book b WHERE b.title = :title", Book.class);

       q.setParameter("title", title);

       return q.getSingleResult();

   }


   @Override

   public Book saveBook(Book b) {

       if (b.getId() == null) {

           em.persist(b);

       } else {

           b = em.merge(b);

       }

       return b;

   }


   @Override

   public void deleteBook(Book b) {

       if (em.contains(b)) {

           em.remove(b);

       } else {

           em.merge(b);

       }

   }

}

Advantages of Repository Pattern
The pattern abstracts the Data Store and enables you to replace your Data Store without changing your business code.

The repository improves the reusability of your database operations code, especially your queries, by encouraging you to implement all the operations in one place. That makes them easy to find and to reuse.
